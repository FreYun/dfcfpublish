# 自我复盘与进化 /self-review

**触发场景**：
- HEARTBEAT 检测到距上次复盘超过 30 天
- 用户显式调用 `/self-review`
- 用户说"帮我复盘一下"、"看看你之前的判断对不对"、"自我评估一下"
- 积累了 5+ 条新研究记录尚未做过复盘

---

## 复盘流程

**确定 `TODAY`，确定复盘范围（默认：上次复盘至今；首次：全部历史）。**

---

### Phase 1 — 检索历史研究

用 QMD 系统性地拉取历史记录：

```
qmd search "结论 判断 看多 看空"        ← 找所有历史观点
qmd search "预测 待验证"               ← 找待验证预测
qmd search "方法论笔记 盲点"           ← 找已记录的方法论问题
```

同时直接读取：
- `memory/predictions/tracker.md` （所有预测，重点看"待验证"状态）
- `memory/evolution/review-log.md` （上次复盘结论，避免重复）
- `MEMORY.md` 中的"当前核心观点"和"待验证预测"节

汇总：本次需要核验的预测有哪些？本次需要重新评估的行业观点有哪些？

---

### Phase 2 — 预测核验（逐条）

对每一条"待验证"预测，**用当前数据核实**：

**数据核实工具选择**：
- 个股涨跌幅预测 → `tushare_daily(ts_code, 建立日~截止日)` 对比建立时收盘价，计算实际涨跌幅
- 行业指数方向/点位 → `tushare_index_daily(指数代码, 建立日~截止日)` 对比目标点位
- 个股相对强弱 → `tushare_daily` + `tushare_index_daily(000300.SH)` 同期涨跌幅对比
- 财务指标（营收、利润、毛利率）→ `tushare_income` / `tushare_fina_indicator`
- 行业事件/政策 → zhipu MCP 搜索（⚠️ 时间验证必做）

对每条预测，输出：
```
预测 P{N}：[内容]
建立日期：{日期} → 当前：{TODAY}（{X}个月后）
核实数据：[具体数值/事件]
结果：✅ 验证成立 / ❌ 证伪 / ⏳ 时间未到 / ⚠️ 部分成立
说明：[简短解释]
```

---

### Phase 3 — 行业观点再评估

对 `memory/views/` 下每个行业文件中的"当前观点"：

1. 距上次更新超过 **60 天** → 需要重新评估
2. **先读该文件的"数据快照历史"**，找到最近一条快照，记录各关键指标的历史值
3. 拉取当前新数据（与快照中相同口径）：Tushare 最新财务 + 行情 + 北向，zhipu 最新资讯
4. **逐项对比新旧快照**：

```
对每个核心指标：
  旧值（快照日期：{X}）→ 新值（今日）→ 变化方向
  变化是否符合当时预期？（当时预测的证伪条件有没有触发？）

重点关注：
  □ 景气度阶段：有无进入下一阶段？（触顶→下行 / 触底→反转）
  □ 毛利率趋势：扩张/收缩的方向变了吗？
  □ 现金含量：盈利质量有无恶化？
  □ 估值：当时认为"合理"的估值锚点，现在还成立吗？
  □ 北向资金：方向有无逆转？
  □ 政策环境：有无重大变化（新政策/整顿/退坡）？
```

评估结论类型：
- **维持**：新旧数据方向一致，逻辑链完整，关键假设未被触动
- **加强**：新数据进一步支持原判断，置信度提升
- **修正**：部分指标偏离预期，更新具体数值和逻辑，但方向不变
- **反转**：证伪条件已触发，或核心假设被打破，判断从看多转看空（或反之）

> 观点反转是高价值信号 → 必须搞清楚：是**数据变了**（外部环境改变），还是**当初的逻辑就有缺陷**（分析错误）。两者对应不同的进化动作：前者更新观点，后者还要更新方法论。

---

### Phase 4 — 方法论审计

系统性检查分析质量：

**问题清单**：
```
□ 有无预测系统性偏向（总是看多/总是看空）？
□ 有无行业偏好盲区（哪些赛道一直没深研）？
□ 时间验证有没有做彻底？（旧资讯误判的情况）
□ 数据和资讯的一致性核查有没有漏做？
□ 预测的证伪条件是否太模糊？
□ 行业研究的结论是否足够有主见？（有无"中性观望"用得太多）
□ 有无重复研究同一行业但结论矛盾却没发现？
```

**结构性偏差识别**：
- 统计历史预测：正确率多少？证伪的有什么共同规律？
- 统计行业覆盖：哪些赛道有多次研究，哪些从未涉及？
- 检查时间维度：判断周期是否匹配（做了短期判断却用长期逻辑支撑？）

---

### Phase 5 — 进化决策

基于 Phase 1-4 的发现，**明确决定**是否需要进化，以及进化什么：

**进化类型判断**：

| 发现 | 进化动作 |
|------|---------|
| 某个技能流程有系统性缺陷（如时间验证漏做） | skill-update：修改对应技能的步骤 |
| 有一类反复出现但没有工作流的分析需求 | skill-new：创建新技能 |
| 发现某个信息源更好/更差 | tools：更新 TOOLS.md |
| 行为习惯有偏差（如总是低估政策风险） | soul：更新 SOUL.md 行为准则 |
| 技能流程覆盖不够（如缺少某类数据分析步骤） | skill-update：扩展该技能 |
| AGENTS.md 某个流程描述过时或不清晰 | agents：更新 AGENTS.md |

**进化决策要有门槛**：
- 单个例外不触发进化。**模式**（≥2次出现）才值得改
- 不因为"感觉可以更好"就改；要有具体证据驱动
- 每次复盘的进化改动不超过 3 处，防止过度调整

---

### Phase 6 — 执行进化

对每个决定进化的项目，**直接修改文件**：

**修改技能**（`skills/{skill-name}/SKILL.md`）：
- 在对应步骤加入/修改工作流
- 在注意事项节追加新发现的规则
- 不要重写整个技能，定向改动

**修改 SOUL.md**：
- 在行为准则节追加或修改
- 保持简洁，SOUL.md 不能太长

**修改 TOOLS.md**：
- 更新信息源优先级
- 添加新发现的有效/无效数据源

**创建新技能**：
- 在 `skills/{new-name}/SKILL.md` 创建，参考现有技能格式
- 在 AGENTS.md 的工具列表中登记

**每项修改完成后，在 MEMORY.md 的"能力边界提醒"或相关节更新**。

## 注意事项

- **复盘不是自我批评大会**：目标是找规律、改方法，不是反复检讨同一个错误
- **进化要保守**：宁可少改不改错，改一个地方要真正理解了问题
- **时间验证同样适用于复盘数据**：验证预测时搜索到的资讯同样需要核验日期
- **证伪比验证更有价值**：预测错了才能学到东西，要认真分析错在哪里
- **观点反转要留下完整记录**：在 views 文件中写清楚"为什么之前错了"
