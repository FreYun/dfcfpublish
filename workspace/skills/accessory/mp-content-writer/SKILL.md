# 公众号投资文章创作 Skill

基于 `writing-styles` 叙事风格库的公众号投资内容创作框架，集成 `fact-check` 数据核查流程。

---

## Step 1: 选类型

先问清楚要写哪种类型，然后 **Read 对应的类型文件**：

| 类型 | 篇幅 | 加载文件 |
|------|------|---------|
| **A. 日复盘** — 交易日收盘后，当天市场回顾+观点 | 1500-2500字 | `Read type-daily.md` |
| **B. 深度分析** — 行业/策略/方法论的深度长文 | 3000-5000字 | `Read type-deep.md` |
| **C. 周复盘** — 每周固定发布的结构化复盘 | 2000-3000字 | `Read type-weekly.md` |
| **D. 轻松深度** — 用段子/故事包装的专业分析 | 1500-3000字 | `Read type-casual-deep.md` |

> 加载类型文件后，按其中的「素材采集」→「写作框架」顺序执行，完成后回到本文件继续 Step 2。

---

## Step 2: 风格指南

### 成文格式铁律

> **最终成文中禁止出现结构标记。**类型文件中的框架标签（如"一句话概览""主题段""引子段""结论段"等）仅用于写作时的结构指引，**不得出现在最终输出的文章正文中**。成文时用自然过渡衔接各部分，读者不应看到任何方括号标签。

### 通用风格要求

| 维度 | 要求 |
|------|------|
| **语气** | 专业但不端着，像一个懂行的朋友在聊天 |
| **用词** | 专业术语 + 口语化表达混搭，关键术语用大白话翻译一次 |
| **句式** | 短句为主，长分析拆成多段，制造阅读节奏感 |
| **观点** | 必须有独立判断，不做复读机；对了不吹，错了不删 |
| **数据** | 关键论点必须有数据支撑，标注数据来源 |
| **诚实** | 不懂就说不懂，比装懂更有信任感 |
| **留余地** | 用"大概率""如果没意外"软化判断，不把话说死 |

### 人设与语气

公众号文章以**真人博主**视角写作，**不使用小奶龙/本龙/龙相关的任何人设元素**。具体要求：
- 第一人称用"我"，但读起来像一个真人投资博主在写，不是一只龙在写
- **禁止出现**："本龙""小奶龙""奶龙""龙""卖萌""小富龙"等内部人设词汇
- **禁止**在文章中使用奶龙式语气（如撒娇、卖萌、自称龙等）
- 语气参考：专业但不端着，有观点有态度，像一个懂行的朋友在分享
- 如果账号的公众号人设尚未明确，先与用户确认

### 绝对不要

- 不要用 Markdown 加粗（`**`）— 公众号正文不需要 Markdown 格式标记，用纯文本写
- 不要写"众所周知""首先让我们来看" — 废话
- 不要纯罗列数据不给观点 — 那是数据终端不是文章
- 不要每天都看多或都看空 — 墙头草没人信
- 不要用"震惊！""暴涨！""重磅！" — 低质标题党
- 不要抄别人的观点当自己的 — 引用要标注来源
- 不要给具体个股买卖建议 — 合规红线

### 通用写作技巧

> 详见 `skills/writing-styles/SKILL.md`，以下为核心要点。

**1. 表格→叙事转化**

| 场景 | 研报写法 | 文章写法 |
|------|---------|---------|
| 价格变动 | `硅料 40元/kg -50% 底部震荡` | "硅料从300块跌到40块，跌了87%——比2015年钢铁去产能还狠" |
| 多指标对比 | 5列×8行大表格 | 只挑最有故事性的2-3个数据，用对比句串联 |
| 公司排序 | `第一梯队/第二梯队` 表格 | "如果只能选一家，选XX——因为..." |

全文表格最多1个（横向对比 ≥ 4家时可用）。

**2. 引用自然化（保留出处）**

| 研报式 | 文章式 |
|-------|--------|
| 据浙商机械（2026-03-30）研报指出 | 浙商机械最新研报提到 |
| 东吴计算机（2026-03-22）研报认为 | 根据东吴计算机的跟踪 |
| 据天风电新（2026-03-30）研报测算 | 天风电新给了一笔账 |

- 保留券商名称，不写具体日期
- 引用后紧跟自己的解读
- 完整出处（券商、标题、日期）交付时以独立清单附给研究部

**3. 开头 Hook 公式**

| 类型 | 模板 | 示例 |
|------|------|------|
| 反直觉数据 | "XX从A到B，[类比]" | "硅料从300跌到40——比房价崩盘还狠" |
| 人物驱动 | "[名人]又搞事了——这次是[行业]" | "马斯克又搞事了——这次是光伏板" |
| 灵魂拷问 | "如果只能选一个方向，你选什么？" | "光伏里只能下一个注，你选哪个环节？" |
| 反差场景 | "A在[惨状]，B在[盛况]" | "硅料厂在裁员，设备商在加班" |
| 时间锚定 | "[时间]前还是[旧]。现在，[新]" | "两年前光伏是最爱赛道，现在基金经理摇头" |

**4. 自我验证**
引用自己之前文章的判断，展示逻辑链条的连贯性。对了就说"之前聊过"，错了就复盘为什么错。

## Step 3: 优化标题

标题决定了 80% 的打开率。写完正文后，用以下清单优化标题：

### 标题自检清单

- [ ] 能引发好奇心吗？（读者看到标题会想"为什么？"）
- [ ] 有没有透露太多？（标题把结论说完了就没人点进来了）
- [ ] 像人说的话吗？（不像论文标题、不像新闻标题）
- [ ] 有情绪共振吗？（读者看到会心一笑/皱眉/好奇）
- [ ] 标题 6 字以内？

### 好标题示例

| 类型 | 举例 |
|------|------|
| 口语反问 | "衰退来了？" |
| 反直觉 | "红利是假的" |
| 情绪短句 | "拐点。。""苟住！" |
| 悬念 | "谁在撒谎？" |
| 判断型 | "油价要涨了" |

## Step 4: 质量自检

完稿后，逐条检查：

### 内容质检

- [ ] 有独立观点，不是复读别人的话
- [ ] 关键论点有数据/事实支撑
- [ ] 逻辑链条完整，不存在跳跃
- [ ] 没有给具体个股买卖建议（合规红线）
- [ ] 涉及投资内容有风险提示/免责声明
- [ ] 没有抄袭，引用有标注

### 表达质检

- [ ] 读出声来像人说的话，不像 AI 写的
- [ ] 开头 3 行能留住人
- [ ] 没有废话段落（每段都有信息量）
- [ ] 短句为主，阅读节奏流畅
- [ ] 标题通过了上面的自检清单

### 格式质检

- [ ] 标题长度合适
- [ ] 正文有分段，不是一坨
- [ ] 关键数据/观点有加粗或其他强调
- [ ] 结尾有收束，不是突然断掉

---

## Step 5: 数据核查

> **铁律：成文后、交付前，必须对文章中的所有数据和事实性表述进行核查，不可跳过。**

### 5.1 提取数据清单

逐段扫描文章，提取所有事实性表述：
- 具体数字（价格、涨跌幅、产量、库存、渗透率等）
- 时间性表述（"本周""2026年一季度"等）
- 事件性表述（"某公司宣布XXX"等）
- 排名/对比性表述（"最低""第一"等）

### 5.2 逐条核查

按以下优先级选择核查手段：

| 优先级 | 核查方式 | 适用场景 |
|--------|---------|---------|
| 1 | research-mcp 工具 | 行情、基金、宏观数据等有对应工具的 |
| 2 | `research_search` / `news_search` | 研报观点、行业数据、事件类表述 |
| 3 | `deepresearch` skill | 需要深度搜索的复杂表述 |
| 4 | 网页搜索 | 公开新闻、公告类事实 |

核查要点：
- **每一个具体数字都必须核查**——价格、百分比、金额、日期，无一例外
- **无法验证 ≠ 正确**——查不到来源的数据标为「待确认」，不能默认正确
- 数据有时效性，确认引用数据对应的时间窗口
- 百分比要核实计算过程（基数和结果是否匹配）
- 转引数据追溯到原始来源

### 5.3 输出核查报告

用以下格式输出，按核查结果分组：

```markdown
## 数据核查报告

### 已验证
| 原文表述 | 核查结果 | 数据来源 |
|---------|---------|---------|

### 有偏差
| 原文表述 | 实际数据 | 偏差说明 | 数据来源 |
|---------|---------|---------|---------|

### 待确认（无法验证）
| 原文表述 | 核查尝试 | 建议 |
|---------|---------|------|
```

### 5.4 修正文章

根据核查结果修正文章：
- 数据有误 → 替换为正确数据并标注来源
- 无法验证 → 删除、模糊化处理（改为趋势描述）、或标注为"市场预期"
- 计算错误 → 修正计算

> 核查报告和修正后的文章一并展示给用户确认。

---

## Step 6: 交付发布

### 6.1 展示文章 + 封面建议

将完整文章（标题 + 正文 + 风险提示）展示给用户，**同时主动附上封面配图建议**，不要等用户要求。

展示内容：
- 文章类型和字数
- 核心观点概述（1-2 句话）
- **封面配图方案**（必做，不可跳过）：
  1. Read `skills/mp-cover-art/SKILL.md`
  2. 分析文章主题和情绪 → 选模板 → 给出 2-3 个封面方案推荐
  3. 让用户一次性确认文章内容 + 选择封面方案

### 6.2 用户确认

等待用户对文章内容和封面方案的确认。确认后：
- 生成封面图，保存到 `workspace/images/cover/YYYY-MM-DD/`
- 如有修改意见则调整后重新展示

### 6.4 发布方式

确认后按用户指定的方式发布：

| 方式 | 操作 | 说明 |
|------|------|------|
| **浏览器发布** | 用 browser 工具（`profile: "bot11"`）打开微信公众平台后台，手动创建图文并粘贴内容 | 需已登录公众号后台 |
| **用户自行发布** | 将文章以 Markdown 格式保存到 `review-output/` 目录，告知用户文件路径 | 用户自行复制到公众号后台 |
| **保存草稿** | 将文章保存到 `memory/公众号草稿/` 目录待后续发布 | 适合需要多篇文章排期的情况 |

> **浏览器发布公众号的详细流程待用户确认后沉淀为独立 skill（`mp-browser-publish`）。** 在此之前，优先使用"用户自行发布"方式。

### 6.5 记录

发布后记录到当日日记 `memory/YYYY-MM-DD.md`：
- 文章标题、类型、字数
- 发布平台和时间
- 使用的数据来源和研究工具
- 写作过程中的经验或教训
