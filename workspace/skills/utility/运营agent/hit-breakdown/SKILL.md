# 🔍 爆款拆解

> 不是"抄爆款"，是"读懂爆款的结构"。一篇帖子火了，不是因为运气，是因为它在标题、开头、情绪、互动设计上同时做对了好几件事。拆出来这些"做对的事"，才能复用。

---

## 核心理念

爆款的底层特征是**跨领域通用**的。理财帖和美妆帖的话题不同，但让人点进去的标题钩子、让人读完的情绪曲线、让人评论的互动设计，底层逻辑是一样的。

**本技能不关心"写什么"，只关心"怎么写才有人看"。**


## 分析方法论

### Step 1 — 单篇拆解（可选）

> 批量数据用下方"批量分析指南"更高效。单篇拆解适合日常发现好帖子时深度学习。

对单篇帖子做 **6 维度结构化拆解**。

#### 拆解卡片模板

```markdown
## 拆解卡片 #{序号}

**基础信息**
- 标题：{原标题}
- 作者：{昵称}（粉丝量 {N}）
- 类型：{图文 / 视频}
- 数据：👍{N} 💬{N} ⭐{N}
- 领域：{理财 / 生活 / 职场 / 情感 / ...}
- 链接：{笔记ID}

---

### 维度一：标题

- **钩子类型**：{痛点 / 反直觉 / 数字 / 悬念 / 身份认同 / 情绪共鸣 / 争议}
- **字数**：{N} 字
- **情绪词**：{提取标题中的情绪关键词}
- **分析**：{为什么这个标题让人想点——一句话}

### 维度二：开头（前 3 行）

- **抓人方式**：{故事开场 / 直接亮观点 / 制造冲突 / 自嘲 / 提问 / 场景代入}
- **原文摘录**：
  > {帖子前 3 行原文}
- **分析**：{开头做对了什么}

### 维度三：结构

- **结构类型**：{故事型 / 清单型 / 对比型 / 自述型 / 教程型 / 争议型 / 反转型}
- **段落数**：{N} 段
- **骨架**：
  1. {第一段在干什么——引入/铺垫/...}
  2. {第二段在干什么——展开/转折/...}
  3. {第三段...}
  ...
- **分析**：{结构上的亮点}

### 维度四：情绪曲线

- **曲线类型**：{开高走平 / 低开高走 / 过山车 / 平铺有转折 / 持续升温}
- **关键转折点**：{在第几段，什么事件/话语触发了情绪变化}
- **分析**：{情绪设计做对了什么}

### 维度五：互动设计

- **结尾钩子**：{提问 / 征集经历 / 抛争议 / 开放结局 / 晒自己等结果 / 无}
- **评论区特征**：{评论主要在聊什么——共鸣 / 争论 / 补充经历 / 求教程}
- **高赞评论摘录**（前 3 条）：
  1. "{评论内容}" — 👍{N}
  2. ...
- **分析**：{为什么这篇评论多}

### 维度六：视觉与排版

- **封面风格**：{纯文字 / 实拍 / 拼图 / 截图 / 手写体 / ...}
- **正文排版**：{密集长文 / 短段落 / 带emoji分隔 / 列表式 / ...}
- **emoji 密度**：{高 / 中 / 低 / 无}

---

### 一句话总结

> {这篇帖子最值得学的一件事是什么}

### 可提炼模式

- {标题模式：xxx}
- {结构模式：xxx}
- {互动模式：xxx}
```

### Step 2 — 模式归纳

> 单篇拆解是素材，模式归纳才是资产。5 篇以上同类拆解才值得归纳为一个"模式"。

**归纳流程**：
1. 通读 `samples/` 下所有拆解卡片
2. 按 6 个维度分别聚类：哪些帖子在某个维度上用了类似手法？
3. 出现 ≥3 次的手法 → 提炼为一个"模式"
4. 每个模式写入 `patterns.md`

**模式的结构**：

```markdown
### 模式名：{简短命名}

> {一句话定义这个模式}

**适用场景**：{什么类型的内容适合用这个模式}

**案例**：
1. 《{帖子标题}》— 👍{N} 💬{N}
   > {关键摘录}
2. 《{帖子标题}》— 👍{N} 💬{N}
   > {关键摘录}

**反面案例**（做不好是什么样）：
> {常见的失败版本}

**使用要点**：
- {要点1}
- {要点2}
```

## 与现有 skill 的集成

### → draft-review（审稿）

Phase 3 标题/内容打磨时，**只读 `patterns.md` 顶部的速查表**（不要读全文）：
- 按问题类型在速查表中找到匹配的模式 ID
- 需要具体案例时，再按 ID 搜索详情部分

审稿意见从"我觉得标题不行"变成"你的标题是信息摘要型（F-02），试试 T-02 反常识断言"。

### → content-calendar（选题日历）

周度规划时参考：
- 最近哪类结构/情绪模式在跑量 → 排期时倾向这类

今日选题.md 可附带推荐参考模式：

## 注意事项

1. **不要复制内容**：拆解是提炼结构和手法，不是搬运正文。patterns.md 里的案例只保留关键摘录
2. **跨领域优先**：不要只采自己 bot 的垂类。美食帖的标题技巧可能比理财帖更值得学
3. **数据会骗人**：高赞不等于值得学。有些帖子纯靠颜值/争议，结构上没什么可学的——跳过
4. **模式会过时**：小红书的算法和用户口味在变，半年前的爆款模式现在可能已经失效。patterns.md 要标注观察时间
5. **量变到质变**：前几次采样可能看不出规律，坚持积累到 30+ 篇拆解后模式会自然浮现

## 批量分析指南

> 当面对 Spider 爬取的大批量样本（50+ 篇）时，逐篇出拆解卡片不现实。以下是批量处理的实操建议。

### 文件结构

```
hit-breakdown/
├── SKILL.md             ← 本文件（方法论 + 批量指南）
├── skill.json
├── patterns.md          ← 通用模式库（核心输出，持续迭代）
├── samples/             ← 原始爬取数据
│   ├── YYYY-MM-DD.md          ← 人可读的采样存档
│   └── YYYY-MM-DD_raw.json    ← 结构化数据（分析主数据源）
└── summaries/           ← 每批次分析摘要
    └── YYYY-MM-DD.md
```

### 阶段一 — 鸟瞰（元数据扫描）

**优先读 `_raw.json`**（结构化、可精准切片、省 token），`.md` 文件作为人可读存档。

提取所有帖子的：标题 + 互动指标 + 粉丝层级 + 关键词 + 话题标签 + 开头前 3 行。这些元数据通常一次能读完。

**产出**：
- 各关键词 × 粉丝层级的互动数据分布
- 标题钩子类型初步聚类（痛点 / 反直觉 / 数字 / 悬念 / 身份认同 / 情绪 / 争议）
- 标记 25-40 篇需要深读的帖子（选择标准：素人爆款优先、互动异常高的、标题/开头看起来有结构特点的）

### 阶段二 — 深读（分批阅读正文）

按阶段一的聚类或关键词分批，每批 8-12 篇，读完整正文。

**分析方式**：
- 对每批用 6 维度框架分析，但**不逐篇出卡片**，而是直接提炼该批次的共性 + 亮点
- 摘录最有代表性的原文片段作为案例素材
- 对比同批次内的高低分帖子，找出差异

> 如果某篇帖子特别有意思、值得留存完整分析，可以额外补一张拆解卡片到 samples/。

### 阶段三 — 归纳（写入 patterns.md）

- 出现 ≥3 次的手法 → 提炼为一个模式
- 每个模式附上具体案例（摘录原文，标注帖子标题和粉丝层级）
- 素人案例排在前面（内容力 > 影响力）
- 写 `summaries/YYYY-MM-DD.md` 记录本次分析的关键发现

**灵活度说明**：
- 以上是建议流程，可以根据数据量和分析目标自行调整
- 数据量 < 20 篇 → 直接用 Step 2 的单篇拆解卡片更合适
- 分析维度不限于 6 个——如果发现"人设代入感"、"争议构造"等新维度很重要，直接加
- patterns.md 的格式也不固定——有的模式适合列案例+公式，有的适合对比分析，自己判断

## 增量更新机制

> patterns.md 是活文档，每次新数据到来时增量迭代，不从头重写。

### 更新流程

1. **读取上下文**：现有 `patterns.md` + 最近一次 `summaries/` 摘要
2. **分析新数据**：对新批次执行鸟瞰 → 深读 → 归纳
3. **对比新旧**：

| 情况 | 处理 |
|------|------|
| **强化** — 新证据支持已有模式 | 补充新案例，更新证据计数 |
| **细化** — 发现子变体或适用边界 | 拆分子模式，或给已有模式加注适用条件 |
| **挑战** — 新数据与已有模式矛盾 | 标注争议，保留两种证据，等更多数据判断 |
| **新增** — 出现全新手法（≥3 次） | 新增模式条目 |
| **老化** — 已有模式在连续 3+ 批次中未观察到 | 标注"待验证"，不急着删除 |

4. **写增量摘要**：`summaries/YYYY-MM-DD.md`，记录本次新发现、强化了哪些模式、有什么争议

### 维护节奏

| 动作 | 频率 | 说明 |
|------|------|------|
| 批量分析 | 每次新数据到来 | Spider 爬完就分析 |
| 模式清理 | 每月一次 | 清理标注"待验证"的过时模式 |
| 维度审视 | 每季度一次 | 回顾 6 维度是否还够用，是否需要新增或合并 |
