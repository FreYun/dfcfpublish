# 每日盘面复盘（daily-market-recap）

> 交易日每日一篇，固定结构，建立读者"每天来看一眼"的习惯。

收到写复盘的指令后，**第一件事是依次读完以下文件**，全部读完再开始任何写作：

```
1. Read skills/daily-market-recap/每日复盘-写作指南.md    ← 结构、铁律、自检清单
2. Read skills/daily-market-recap/行情数据获取.md          ← 取数流程
3. Read skills/daily-market-recap/券商观点获取.md          ← 搜索当日券商策略观点
4. Read skills/daily-market-recap/基金推荐规则.md          ← 基金筛选脚本调用方式
5. Read skills/daily-market-recap/引用来源管理.md          ← 观点归因规则
```

**没读完这 5 个文件就动笔 = 违规。** 不许凭记忆写，每次都要重新读。

## 对标格式

**东方财富证券每日复盘推送**，四段式：

| 区域 | 内容属性 | 说明 |
|------|---------|------|
| 01 重磅资讯 | 事实（摘录） | 研究部提供链接，bot 选取关键段落 |
| 02 数据复盘 | 配图建议 | bot 写建议，研究部后期插图 |
| 03 盘面解读 | 分析 | bot 自己写，400-600 字，融入券商专业分析 |
| 04 后市展望 | 观点 | bot 自己写，800-1200 字，必须参考当日券商策略观点 |

## 基金筛选执行顺序（必须先做）

当你进入 04 后市展望、需要给某个板块/主题匹配基金时，**不要自己在基金池里人工挑，也不要先凭记忆写基金推荐**。必须先调用本地 Python 脚本，拿到结构化结果后，再继续写下一步。

### 环境变量配置（首次使用必看）

基金筛选脚本依赖外部 LLM API，**首次调用前必须确认以下环境变量已设置**：

| 环境变量 | 必填 | 说明 |
|---------|------|------|
| `FUND_SELECTOR_API_URL` | ✅ | LLM API 地址（如 OpenAI 兼容接口的 `/v1/chat/completions`） |
| `FUND_SELECTOR_API_KEY` | ✅ | LLM API Key |
| `FUND_SELECTOR_MODEL` | 否 | 模型名称，默认 `kimi-k2.5` |
| `FUND_SELECTOR_TIMEOUT` | 否 | LLM 请求超时秒数，默认 60 |

如果未设置必填项，脚本会报错提示缺少哪个环境变量。请联系研究部获取 API 配置。

### 固定执行方式

先确定你要推荐的 2-3 个板块/主题，然后直接运行：

```bash
python skills/daily-market-recap/fund_selector_llm.py --topics 科技,新能源,医药
```

单个主题也可以这样调用：

```bash
python skills/daily-market-recap/fund_selector_llm.py --topic 创新药
```

如需看候选和分层细节：

```bash
python skills/daily-market-recap/fund_selector_llm.py --topic 创新药 --debug
```

### 执行铁律

1. **先跑脚本，再写基金推荐文案**
2. **脚本输出出来之前，不要自己手选基金**
3. **脚本输出后，再根据返回结果写展示文本**
4. **如果脚本返回空结果或报错，先处理筛选问题，不要跳过这一步直接继续写稿**
5. **不要绕过这个脚本直接去读 CSV/XLSX 手工筛基金**

### 你要如何使用脚本结果

脚本会返回 JSON，其中最重要的是：
- `results[].selected_funds`
- `results[].display`
- `display_lines`

拿到结果后：
- 先检查每个主题是否真的有 2-3 只基金
- 再把 `display` 或 `display_lines` 融入 04 后市展望末尾
- 然后再做后续写作和排版

### 为什么必须这样做

因为基金筛选已经收敛到这个 Python 脚本里：
- 它会读取固定的 3 个基金文件
- 它会调用大模型做主题匹配
- 它会按优先级补位
- 它会输出可直接使用的基金推荐结果

所以，**筛选基金这一步的标准动作，就是先调用 `fund_selector_llm.py`，等输出，再做下一步。**

## ⛔ 最高铁律：新闻只许截取，禁止生成

**01 重磅资讯中的所有新闻内容：**

1. **每条都要摘录，不许漏条** — 研究部给了几条新闻，01 就必须出现几条，不许擅自删掉任何一条
2. **只许截取，不许生成** — 01 区域的每一句话都必须来自研究部提供的链接原文。禁止自己编写、总结、归纳、生成任何新闻内容
2. **每条至多 2 段** — 优先摘新闻事实，可适当带原文中的解读，但重心在事实本身
3. **只许摘录，不许改写** — 选中的段落保持原文不变，不做任何改写、缩写、合并、意译
4. **不许杜撰** — 禁止编造新闻事实、数据、人物发言。没有可靠信源，宁可不写
5. **不得改变语义** — 不得通过截取、拼接、省略改变原文语义和倾向
6. **01 区域不加主观评价** — "利好""利空""重磅"等定性词不得出现在 01 区域，解读放 03/04

> 选哪段你来定，但选了就不许改。自己写的内容不许放进 01。

### 行情时效性例外

新闻中提到的具体行情数据（价格、涨跌幅、点位等）可能已过时。摘录前**必须先用浏览器核实最新行情**：

- 验证渠道：[东方财富](https://www.eastmoney.com)、[Mysteel](https://www.mysteel.com)、[金十数据](https://www.jin10.com)
- 如果最新行情与新闻中的数据**差异不大**，照常摘录原文
- 如果最新行情**已出现较大变化**（例如新闻说"油价大涨5%"但此刻已回落），则：
  1. **跳过已过时的行情描述段落**，不摘录
  2. **改为选取**原文中与最新行情逻辑一致的其他段落（如事件背景、政策动态等）
  3. 选中的段落**仍然不许修改**，摘录铁律不变
- 本条不是允许改写的例外，而是要求**避开过时信息、选取仍有效的内容**

## 浏览器使用规范

- **禁止爬虫**（curl/requests/WebFetch），必须用浏览器访问链接
- **禁止擅自启动有头浏览器**（headed browser / `-headless=false`），未经研究部同意不得使用
- 读完链接后**必须关闭浏览器**

```
browser_navigate(url: "链接")
browser_snapshot()
browser_close()   ← 读完必须关闭
```

## 研报引用规则

03 盘面解读和 04 后市展望中如需引用研报、券商观点等资讯：

- **当日研报优先**：有当天发布的研报/观点，优先采用，时效性最强
- **历史研报须严格甄别**：引用非当日研报时，必须判断其核心观点是否已被后续行情或政策变化推翻。如果市场环境、数据面已发生实质性变化，该观点视为过时，**禁止引用**
- **过时判断标准**：研报的关键假设（如利率走向、供需格局、政策预期）与当前实际情况不符，即为过时
- 引用时注明研报机构和日期，让读者自行判断时效

## 合规红线

- **不给操作建议** — 不说"建议买入/卖出""建议抄底""建议减仓""不建议追涨"等任何指导读者具体操作的话
- 不说"配置方向""建议配置"，用"重点关注""看好方向""值得留意"
- 不说"必涨/必跌/稳赚"，用"存在机会/需要警惕"
- 不提个股代码/名称，用行业板块描述（基金代码/名称除外，允许出现）
- 不做收益承诺，不炫耀具体盈亏数字
- 文末必加风险提示

## 写稿完成后

1. **存档**：将完整稿件保存到 `memory/daily-recap/YYYY-MM-DD.md`（以当日日期命名）。文件内容包含完整标题和正文，顶部加一行 `# 每日复盘 YYYY-MM-DD`。如当日已有同名文件则覆盖（以最新版本为准）。
2. **数据核查（必做）**：存档后、呈审前，**必须执行 fact-check skill** 对完整稿件进行数据核查。读取 `skills/fact-check/SKILL.md` 并按其流程逐条核实文中所有数字、事实性表述。核查发现偏差或待确认项时，先修正稿件再进入下一步。**未通过数据核查不得呈审。**
3. **呈审**：数据核查流程和完整稿件**必须先呈给研究部确认**，获得明确同意后才可交由 **gzh-publish** skill 发布。未经确认不得自行发布。
