# 📝 审稿

> 审稿不是找茬，是帮 bot 看到自己看不到的盲区。好的审稿让 bot 觉得"原来还能这样写"，而不是"又被打回来了"。

---

## ⚠️ 读到这里的你,先看一眼这个翻车故事

**2026-04-14 bot7 沪指 4000 点事件**:用户让 sys4 去指导 bot7 发笔记,sys4 临时指派了选题给 bot7,bot7 写完返稿——**这就是典型的"审稿场景",只是触发路径是"派题回审"而不是"bot 主动交稿"**。实际过程里 sys4 没读本文件,凭着前一天改版讨论的记忆现编了一套 9 项 checklist,输出格式是表格+亮点而不是教练指令,也没落 reviews 记录。全错。

正确打开方式是:识别"sys4 派题回审 = 审稿"→ Read 本文件 → 走轻量档(Phase 1 人感 + Phase 2 Lite 对照当初的指派意图 + Phase 3A 标题)→ 50-100 字出反馈 → 落一行 reviews。

**如果你是正在处理某个 bot 返稿的 sys4**:停一下,先读完下面的"何时使用 / 强度分级"节,再动任何笔。凭记忆 = 重演翻车。

## 定位

| 层级 | 负责人 | 管什么 |
|------|-------|--------|
| 合规层 | compliance-review | 法律红线、平台规则、敏感词 |
| 写作层 | 各 bot 自带的内容风格 skill | 标题、结构、排版、SEO |
| **主编层** | **本技能** | **战略契合度、人感、节奏、差异化** |

本技能审的不是"这篇写得对不对"，而是"这篇发出去对这个账号的长期价值是正的还是负的"。

---

## 何时使用 / 强度分级

> draft-review **不是每次都跑全流程**。判断"该不该审"和"审多重"本身就是主编的职责。不要把每篇草稿都拖到完整三 Phase，也不要凭感觉编一套自己的 checklist——按下面的强度对照执行。

### 触发场景 → 强度

> **完整档的判定方式已变更**：不再要求审稿时 sys4 自己识别"高风险"或"数据下滑"——sys4 在审稿这个时点上既没有时间序列数据也没有合规词库。**完整档的触发信号在 `queue/botN.md` 顶部的「状态告警」块里**，由 content-calendar 在每日 Step 1.5 排期时生成/更新。审稿前**必须读这个块一次**（这是 Phase 0 的强制动作之一），有勾选项就升级到完整档，没有就走轻量。

| 场景 | 触发条件 | 强度 |
|------|---------|------|
| 状态告警勾选「新 bot 前 2 周」 | 读 `queue/botN.md` 顶部告警块 | **完整** |
| 状态告警勾选「风格调整后首周」 | 读 `queue/botN.md` 顶部告警块 | **完整** |
| 状态告警勾选「数据下滑期」 | 读 `queue/botN.md` 顶部告警块 | **完整** |
| 状态告警勾选「高风险窗口」/ 当日 queue 条目带 ⚠️ 高风险标签 / 用户在指派时口头要求严审 | 读 `queue/botN.md` 顶部告警块 + 当日条目 | **完整** |
| 老 bot 排期稿回审 | bot 按 `queue/botN.md` 的当天安排写完交审，且**告警块无勾选** | **轻量** |
| sys4 派题回审 | sys4 临时指派具体选题（非 queue 计划内，通常通过对话直接推给 bot），bot 写完交审，且**告警块无勾选** | **轻量** |
| 老 bot 自选题（非排期） | bot 用 topic-library 内素材自发选题，且**告警块无勾选** | **轻量** |
| 仅标题打磨 | bot 明确说"正文 OK，帮我看下标题" / 同选题正文已审过完整版只改了标题 / sys4 排期时标"标题需重点打磨" | **标题专审** |
| 当日时效快反应 | 字数 < 100 且为当日热点/数据快反应（晚 30 分钟就过气的那种） | **跳过** |

> **告警块缺失怎么办**：如果 `queue/botN.md` 顶部根本没有「状态告警」块（旧文件没有这个段落），说明 content-calendar 还没补上 Step 1.5 的输出。这时 draft-review **保守起见走轻量档**，并在反馈最后追一句"⚠️ 该 bot 的 queue 还没维护状态告警块，建议 sys4 下次排期补上"。**不要假设没有告警块就等于没事**。

### 强度说明

| 强度 | 走哪些 Phase | 输出长度 | 落档要求 |
|------|------------|---------|---------|
| **完整** | Phase 0 → 1 → 2 → 3，每 Phase 出 checkpoint | 100-200 字 | `memory/reviews/YYYY-MM-DD-botN.md` 完整记录 |
| **轻量** | Phase 1 人感快检 + **Phase 2 Lite**（只读参考系那一行/一条，做一次"我的指派意图是否被 carry 出来"的对照，不跑 Phase 2 完整 checklist）+ Phase 3A 标题内容打磨；跳过 Phase 3B/3C。**对照参考系**：排期稿对照 `queue/botN.md` 当日条目那一行，派题回审对照"当初发给 bot 的那条指派消息"，自选题对照 `topic-library.md` 中那条素材 | 50-100 字 | 同路径，**只写一行**："轻量审 / ✅ / 备注" |
| **标题内容专审** | 只跑 Phase 3A 打磨；不对照排期、不动正文。**SOUL 只为判"标题语气是否一致"临时 Read 一次该 bot 的 SOUL「说话风格」节**，读完就丢、不扩散到正文。读 `patterns/INDEX.md` 速查表里的 T 类（titles）即可 | 30-60 字（给 1-3 个候选标题 + 一句"为什么这样改"） | 同路径，**只写一行**："标题专审 / ✅ / 备注" |
| **跳过** | 不介入 | — | **不写**，但要在给 bot 的回复里**显式说明**"这篇我跳过审稿，原因是 X"，不能隐式忽略 |

### 出口契约（**完整 / 轻量 / 标题专审**都必须满足）

1. 输出**必须**包含以下任一标记开头，**不准用自己临时编的 checklist 替代**：
   - 发布前审稿：`✅ 走后续发稿流程` / `🔧 改了再发` / `❌ 换方向`
   - 已发布复盘：`✅ 这篇发得好` / `🟡 这篇下次可以更好` / `🔴 这条路下次别走`
2. **落档** — 写 `workspace-sys4/memory/reviews/YYYY-MM-DD-botN.md`（目录不存在先 mkdir）
3. 🚨 **送达写稿 bot 本人** — 落档之后**必须**额外调用 `send_message` 把审稿结论推给写稿 bot，**两种场景都要做**：
   - 发布前审稿：bot 发草稿过来后在等回音，你**必须 send_message 回去**（openclaw 没有自动 reply 机制，sys4 的 session 输出不会自动流回 bot，不显式发 = bot 永远收不到）
   - 已发布复盘：bot 没在等，但审稿的全部意义就是"让写稿者下次写得更好"，**不 send_message = 审了等于白审**，reviews 文件只是 sys4 自己的档案，bot 不会主动读
   - **推送消息正文（发布前审稿）** = 落档 md 里的结论段 + 一句"详见 `workspace-sys4/memory/reviews/YYYY-MM-DD-botN.md`"
   - **推送消息正文（已发布复盘）** = 落档 md 里的结论段 + 一句"详见 `workspace-sys4/memory/reviews/YYYY-MM-DD-botN.md`" + **一段固定的"沉淀指令"**（见下方）
   - **不做这一步的审稿直接算作废**，reviews 文件也算"只写一半的档"，不满足出口契约

#### 已发布复盘的"沉淀指令"（固定附在 send_message 正文末尾）

已发布的稿子改不了，复盘的价值**必须**落到 bot 的长期记忆里，否则下次 bot 还会重犯同样的错。所以每次已发布复盘的推送消息末尾都要显式指示 bot 把教训 append 到自己的长期教训文件：

```
---
📝 沉淀教训（请 bot 执行）：
读完上面的结论后，请用 Edit/Write 把这次复盘的**一句话教训**追加到 `workspace-self/memory/写稿经验.md`（文件不存在就新建）。追加一条即可，格式：

## {YYYY-MM-DD} 《{本次被复盘的标题}》— {✅ 发得好 / 🟡 下次改 / 🔴 下次别走}

- **一句话教训**：{从上面的"下次改的/下次换路子"里提炼一句，面向"下一次写稿"的具体指令，不是复述问题}
- **对照参考**：{正面参考的旧作标题，如有}

沉淀完就结束，**不用 reply_message**（cron isolated session 无回路）。这份文件是你下次开写前应该先 Read 的长期背景，越积越厚。
```

> **为什么强制沉淀**：reviews 文件只是 sys4 这边的档案，bot 不会主动去读。如果 bot 不把教训写到自己 workspace 下的长期文件里，下一次写稿时它没有这段上下文，复盘就只剩 sys4 的情绪宣泄，真正的写作者不会进步。bot5 自己已经养成了维护 `memory/写稿经验.md` 的习惯（2026-03-29 那条就是例子），这里是把它固化为全 bot 的约定。

> **文件名约定**：统一使用 `workspace-self/memory/写稿经验.md`。bot5 已经在用这个名字，新 bot 第一次收到复盘消息时自动创建。未来 sys4/dashboard 可以批量扫所有 bot 的 `memory/写稿经验.md` 做 cross-bot 模式提炼。
4. 给改写不给评语、给位置不给原则、≤ 200 字（轻量 ≤ 100 字，标题专审 ≤ 60 字）

> **反面教训（2026-04-14 bot7 沪指 4000 点）**：见文件顶部「⚠️ 翻车故事」节。正确打开方式是"sys4 派题回审 → 轻量档"：Phase 1 人感快检 + Phase 2 Lite（读一眼当初的指派消息做意图对照）+ Phase 3A 标题打磨，50-100 字出反馈，落一行 reviews。

## 审稿流程（渐进式加载）

> **核心原则**：每个 Phase 只读当前需要的文件，读完立刻用。每个 Phase 结尾必须输出 checkpoint，不输出不进下一阶段。

---

### Phase 0 — 接活 + 强度判定

**本阶段读取**：`workspace-sys4/queue/botN.md` 顶部的「状态告警」块（块的字段定义见 [content-calendar.md](content-calendar.md) 「queue/botN.md 格式」节），外加当日那一行条目（**仅用于识别 ⚠️ 高风险标签**；角度 / 注意事项 / 参考旧作等字段留给 Phase 2 完整读）。读这两样是为了在进 Phase 1 之前判定本次审稿走哪一档（完整 / 轻量 / 标题专审 / 跳过），不要读 SOUL、不要读 patterns。

```
📋 Phase 0 checkpoint：
- 待审 bot：{bot_name}（{account_id}）
- 草稿标题：{title}
- 草稿类型：{图文/视频脚本/简评}
- 草稿字数：{N} 字
- 状态告警块：{勾选项列表 / "无勾选" / "块缺失"}
- 本次走档：{完整 / 轻量 / 标题专审 / 跳过}（依据上面的告警 + 触发场景表判定）
- 对照参考系：{queue 排期 / 派题消息 / topic-library / 仅标题}
```

**判定规则**：

1. 告警块有任意勾选 → **完整**
2. 告警块空但 bot 主动交稿（无论排期、派题、自选题）→ **轻量**
3. bot 明确只要标题反馈 → **标题专审**
4. 字数 < 100 的当日时效快反应 → **跳过**（在 checkpoint 里说明跳过原因）
5. 告警块**缺失**（queue 文件没这一段）→ 保守走 **轻量**，反馈尾部追"⚠️ queue 缺告警块，建议下次排期补上"

确认无误后进入对应 Phase。**跳过档**直接给 bot 一句 ✅ + 跳过原因，不进 Phase 1。

---

### Phase 1 — 人感快检

> 人都不像了，其他都没意义。人感是最基础的关卡。

**本阶段读取**（仅这两项）：
1. 该 bot 的 `SOUL.md` — 人设、性格、说话方式
2. 该 bot 装备的**内容风格 skill** — 具体写作规范

**检查清单**：

- [ ] 通读草稿，**闭眼想象**这是谁在说话 — 能对上人设吗？
- [ ] 有没有**该 bot 不会说的话**？（宣妈突然用研报术语、来财妹妹突然很严肃）
- [ ] 情绪是否**真实可信**？（不是永远阳光、永远理性）
- [ ] 有没有 **AI 典型痕迹**？
  - "首先...其次...最后..."
  - "综上所述"、"让我们一起..."
  - "在当前市场环境下"
  - 每段结构整齐、长度均匀
- [ ] 有没有超出人设知识边界的内容？（宣妈不该写半导体产业链分析）
- [ ] **小红书平台格式兼容性**（小红书正文编辑器**不渲染 markdown**，以下写法会以原始字符形式显示在帖子里，必须改成自然表达。这是硬规则，不是建议）：
  - **加粗 `**xxx**` / 斜体 `*xxx*`** → 小红书不识别星号语法，会把星号一起显示出来。想强调就**拆一行**、**加 emoji**、**用破折号/分号的节奏**替代，不要靠粗体
  - **markdown 表格**（`| 列A | 列B |` + `|---|---|` 的行）→ 在小红书里会变成一堆"文字+竖线"乱码，完全破坏排版。必须改写成**自然语言段落**或**多条短列表**（"米兰：今年降三次，支持提前降息；古尔斯比：2026 年后，通胀黏性高，推迟降息更稳妥"）
  - **markdown 标题**（`# 一级` / `## 二级` / `### 三级`）→ 小红书无标题语法，井号会原样显示。用**emoji 打头的小短句**（"🔑 核心观点"、"📌 三个信号"）做视觉层级
  - **代码块 ` ``` ` / 行内 `` ` ` ``** → 小红书不渲染，反引号会原样显示。股票代码/数字直接用纯文本
  - **markdown 列表**（`- xxx` / `1. xxx`）→ 视觉上能接受，但**小红书里更推荐用 emoji 数字**（1️⃣/2️⃣/3️⃣）或符号（🌟/✨/•）替代，观感更符合平台调性

**Gate 判定**：

| 结果 | 动作 |
|------|------|
| 🟢 人感OK & 格式干净 | → Phase 2 |
| 🟡 轻微AI味 / 有零星 markdown 残留（1-2 处加粗/表格） | 记录问题，→ Phase 2（最终意见里会附带改写建议，**必须**点明 markdown 不兼容并给出替换写法） |
| 🔴 严重不像人 / 人设串台 / **大段 markdown 污染**（整篇充斥表格、每段都有加粗、用井号做标题） | **直接打回**，输出问题 + 改写方向，不进 Phase 2。人感问题触发 human-feel-audit（见下）；格式污染不触发 audit 但**必须**在反馈里明确说"小红书不渲染 markdown，全篇要改成自然表达" |

> **格式污染的判定阈值**：≥3 处 `**加粗**`、或**任意一个** markdown 表格、或**任意一处**井号标题 → 计入格式问题必须在反馈里指出；如果这些格式伪装了本该用人话说的内容（比如用表格绕过观点陈述、用加粗堆砌卖点），同时计入人感问题。

> 🚨 **人感 🔴 → 硬触发 human-feel-audit**（不是软建议）
>
> 当 Phase 1 判定 🔴 时，本次审稿任务**未完成**，必须在给 bot 反馈之后追加以下动作（同一会话内完成，不准跨会话拖延、不准只挂 TODO）：
>
> 1. `Read workspace-sys4/skills/content-ops/human-feel-audit.md`
> 2. 按该 skill 的"诊断前准备"采集该 bot 近 14 天帖子，跑完 Step 1 通读 + Step 2 七宗罪逐项 + Step 3 开处方
> 3. 用该 skill 的"诊断报告模板"产出报告，**落档到 `workspace-sys4/memory/human-feel-audits/YYYY-MM-DD-botN.md`**（目录不存在就先 mkdir）
> 4. 在给 bot 的最终反馈末尾附一句："⚠️ 已对你做一次完整人感诊断，详见 `memory/human-feel-audits/YYYY-MM-DD-{你的bot名}.md`，请重点改其中 🔴/🟡 的几条"
>
> **理由**：单次 🔴 表示人感系统性失守，单篇打回只治本次稿、治不了根。完整 audit 是唯一能把"七宗罪里到底中了哪几条"说清楚的手段。审稿场景是 audit 唯一**能被自然勾起**的入口，不在这里勾就永远勾不起来。
>
> **跳过条件**：仅当本会话 24 小时内已对同一 bot 做过 audit（看 `memory/human-feel-audits/` 里有没有今天的文件），可以跳过新一轮 audit，但必须在反馈里说明"今日已 audit，本次不重复"。

```
📋 Phase 1 checkpoint：
- 人感评级：{🟢/🟡/🔴}
- 平台格式：{🟢 干净 / 🟡 零星 markdown 残留（列出位置）/ 🔴 大段污染（列出类型:表格/加粗/标题/代码块）}
- 发现的问题：{列出，或"无"}
- 决定：{继续 / 打回重写 / 打回重写并追加 human-feel-audit}
- 是否触发 audit：{否 / 是 → 落档路径 memory/human-feel-audits/YYYY-MM-DD-botN.md}
```

### Phase 2 — 排期对照

> 选题方向、支柱轮转、撞题检查已由 content-calendar 完成并写入排期队列。审稿只需对照：草稿是否符合排期指导。

**本阶段读取**：
1. 该 bot 的 `今日选题.md`（由 content-calendar 每日推送到 `workspace-botN/`）
2. 如果 `今日选题.md` 不存在，读 `workspace-{self}/queue/botN.md` 查当天排期

> 不需要重新读 SOUL.md 查支柱、不需要 query-stats.sh 查撞题 — 这些 content-calendar 已经做过了。

**检查清单**：

- [ ] 草稿选题是否与排期安排的方向**一致**？（如果 bot 写的跟排期完全不同，需要判断原因）
- [ ] 排期指定的**角度**是否被采纳？（角度可以微调，但不应完全跑偏）
- [ ] 排期中的**注意事项**是否被遵守？（如"上次这个支柱数据差，这次换个方式"）
- [ ] 排期附带的**参考旧作**有没有被借鉴？（风格、结构、情绪处理）

**特殊情况**：
- **bot 没有按排期写（自选题）**：不直接打回。检查该选题是否来自 bot 的 `memory/topic-library.md`，如果是紧急/时效素材且合理，🟢 放行
- **没有排期（空白日）**：回退到基础检查 — 快速确认选题在支柱范围内、近 7 天无严重撞题（`bash dashboard/query-stats.sh {botN} --titles`）

**Gate 判定**：

| 结果 | 动作 |
|------|------|
| 🟢 符合排期 / 合理自选 | → Phase 3 |
| 🟡 偏离排期但选题合理 | 记录偏离原因，→ Phase 3 |
| 🔴 完全跑题且无合理理由 | **打回**，提醒 bot 查看今日选题.md |

```
📋 Phase 2 checkpoint：
- 排期对照：{符合 / 微调 / 偏离 / 无排期}
- 偏离说明：{如有，说明 bot 为什么没按排期写，是否合理}
- 决定：{继续 / 打回看排期}
```

### Phase 3 — 标题内容打磨

> 通过了人感和战略，说明这篇"该发"且"像人"。现在打磨到"能火"。

**本阶段读取**：
1. 该 bot 的 `写稿经验.md`（如有）— 之前踩过什么坑
2. `patterns/INDEX.md`（和本文件同级）— 只读顶部速查表，按问题类型找模式 ID；需要案例时再按 ID 前缀打开对应维度文件（T→titles.md, O→openings.md, S→structures.md, E→emotions.md, I→interactions.md, V→visuals.md, F→failures.md）

#### 3A. 标题吸引力

- [ ] 封面标题 ≤ 20 字？
- [ ] 前 10 字有钩子（痛点/痒点/热点至少命中一种）？
- [ ] 标题有**情绪**吗？（让人想点进来的冲动，不是信息摘要）
- [ ] 放在信息流里，**3秒内能产生点击欲望**吗？
- [ ] 跟该 bot 的人设**语气一致**吗？（完整 / 轻量档凭 Phase 1 已读的 SOUL.md 记忆判断；**标题专审档**在本阶段临时 Read 一次该 bot 的 SOUL「说话风格」节，只用于判这一条）

**标题常见病和药方**：

| 症状 | 你写的 | 别人写的（爆了） | 差在哪 |
|------|--------|-----------------|--------|
| 像研报摘要 | "2026年4月黄金走势分析" | "金价又新高，但我反而想跑了" | 没有"我"，没有立场 |
| 什么都想塞 | "从宏观经济角度看美联储降息对黄金价格的影响" | "降息了，金价却跌了？" | 信息流只露20字 |
| 剧透结局 | "定投三年赚了30%" | "定投三年，聊聊我差点清仓的那个月" | 要悬念不要结论 |
| 人设串台 | 宣妈突然写"从产业链角度看" | "又看不懂了，但我黄金还好好的🪙" | 不懂也是人设的一部分 |
| 自嗨型 | "我来给大家分享一下投资心得" | "亏了两万之后我想明白一件事" | 没人关心你要分享什么 |
| 正确的废话 | "投资要长期持有，不要追涨杀跌" | "拿了半年终于回本，但我决定卖了" | 读者要真人的挣扎和选择 |

#### 3B. 内容质量

- [ ] **开头3秒**：前2-3行是否立刻抓人？（不要"大家好今天聊聊"）
- [ ] **信息增量**：读完后读者学到了什么或感受到了什么新东西？
- [ ] **情绪曲线**：内容有起伏吗？（纯平铺直叙 = 无聊）
- [ ] **结尾钩子**：结尾引导互动了吗？（提问、征集、争议、晒自己）
- [ ] **长度合适**：简评 150-300 字、日常 300-600 字、深度 600-900 字
- [ ] **零废话**：删掉任何一段文章是否缺了什么？不缺 = 那段是废话

**信息增量快速测试**——读完正文，读者会说什么：
- "哦原来是这样" ✅ 有新认知
- "卧槽没想到" ✅ 有反直觉
- "嗯嗯对对" ✅ 有共鸣
- "所以呢？" ❌ 没有增量
- "这不废话吗" ❌ 纯正确但无价值

#### 3C. 互动潜力

> CES 算法中，评论和分享的权重是点赞的4倍。

- [ ] 是否留了**讨论空间**？（话说死了反而没人评论）
- [ ] 是否有**争议性观点**？（温和的正确没人讨论）
- [ ] 结尾的互动引导**是否自然**？（不要硬加"关注+收藏"）
- [ ] 是否有**评论区二创空间**？（读者可以分享自己的经历）

```
📋 Phase 3 checkpoint：
- 标题评级：{🟢/🟡/🔴}
- 内容评级：{🟢/🟡/🔴}
- 互动潜力：{🟢/🟡/🔴}
- 写稿经验命中：{列出踩中的旧坑，或"无"}
```

## 最终输出

**合并所有 Phase 的 checkpoint，直接告诉写稿 bot 该怎么做。**

> 输出对象是写稿 bot，不是人。语气像教练对球员：直接、具体、可执行。不要写分析报告，写行动指令。

### 结论

先看稿子状态:

- **发布前审稿**(草稿还没发,sys4 介入是为了 gate)→ 走下面的「✅ 走后续发稿流程 / 🔧 改了再发 / ❌ 换方向」三选一
- **已发布复盘**(稿子已经发出去,sys4 介入是为了"下次怎么写得更好")→ 走下面的「✅/🟡/🔴 已发布复盘」三档

两类结论都是"一句话标记 + 教练指令",差别只是**用词的时态**:发布前是**前瞻式**("走后续发稿流程 / 改了再发 / 换方向"),已发布复盘是**后视式**("继续这么写 / 下次改这里 / 下次换路子")——不要把"走后续发稿流程"这种前瞻动词用在已经发出去的稿子上,会引发语义错位(sys4 自己都会在 thinking 里自嘲"实际是复盘性质的确认")。

### 发布前审稿的结论三选一

> 🚨 **三档写完后的动作顺序是固定的**:
> 1. Write 落档到 `workspace-sys4/memory/reviews/YYYY-MM-DD-botN.md`
> 2. `send_message` 把结论推送给写稿 bot(bot 发草稿过来后在等你回音,不推 = bot 永远收不到,会一直阻塞)
> 3. 最后才输出 text 回复给当前对话方
>
> 顺序不能颠倒、三步都不能省。

**✅ 走后续发稿流程**（所有维度 🟢）：

```
✅ 这篇可以走后续发稿流程（合规 → 印务局 → 发布）。
{一句话说哪里写得好，让 bot 知道什么该继续保持}
⚠️ 发之前自查一下最近有没有其他号同天发类似话题。
```

**🔧 改了再发**（有 🟡，无 🔴）：

```
🔧 这篇方向对，改几个地方就能发。

要改的：
1. {具体位置}：{现在的问题} → {怎么改}
   改写参考："{具体改写后的文字}"
2. {具体位置}：{现在的问题} → {怎么改}
   改写参考："{具体改写后的文字}"

不要动的：
- {哪里写得好，别改坏了}

改完直接走后续发稿流程（合规 → 印务局 → 发布），不用再回我过审。
⚠️ 发之前自查一下最近有没有其他号同天发类似话题。
```

**❌ 换方向**（有 🔴）：

```
❌ 这篇先别发，{一句话说核心问题}。

问题：{是人设不对 / 选题偏了 / AI味太重，具体说}

建议的方向：
- {方向A}：{为什么这个更契合人设}
- {方向B}（如有）：{另一个可行角度}

可以参考你之前写的《{某篇写得好的}》，那篇的{xxx}处理得很好。
```

### 已发布复盘的结论三档

> 已发布的稿子改不了了,复盘的作用是**让下次写得更好**。结论用后视式措辞,重点是"继续这么写 / 下次改 / 下次换"。
>
> 🚨 **三档写完后的动作顺序和发布前审稿一样**:
> 1. Write 落档到 `workspace-sys4/memory/reviews/YYYY-MM-DD-botN.md`
> 2. `send_message` 把结论推送给写稿 bot(**bot 这次没在等你、不主动读 reviews 文件,不推 = 审了等于白审**) — **⚠️ send_message 正文末尾必须附上出口契约里规定的"沉淀指令"**（见上文「已发布复盘的"沉淀指令"」一节），指示 bot 把教训 append 到 `workspace-self/memory/写稿经验.md`。没有沉淀指令 = bot 看完就忘 = 复盘白做。
> 3. 最后才输出 text 回复给当前对话方(通常是用户)
>
> 顺序不能颠倒、三步都不能省。已发布复盘尤其容易在第 2 步掉链子——因为 bot 没发消息过来,你会"觉得"没人在等,于是跳过送达——错。审稿的唯一价值是让写稿者进步,bot 不收到就是零价值。**且 bot 收到后不沉淀到自己 memory 里,下次写稿依然没有这段上下文,也等于零价值**——所以沉淀指令是第 2 步不可分割的组成部分,不是可选项。

**✅ 这篇发得好**（所有维度 🟢）：

```
✅ 这篇发得不错，下次继续这么写。

做得好的：
- {具体位置/手法}：{为什么好}
- （1-3 条，只说最该保持的，不要平均用力）

下次可以更进一步的（可选）：
- {一条微调建议，没有就不写}
```

**🟡 这篇下次可以更好**（有 🟡，无 🔴）：

```
🟡 这篇可以发，但几个点下次注意。

做得好的：
- {具体位置/手法}：{为什么好}

下次改的：
1. {具体位置}：{这次的问题} → {下次怎么写}
   下次参考："{具体改写示例}"
2. {具体位置}：{这次的问题} → {下次怎么写}

可以对照你之前《{某篇写得好的}》的{xxx}处理，那篇这一点做得比这篇好。
```

**🔴 这条路下次别走**（有 🔴）：

```
🔴 这篇发了就发了，但下次**不要再这样写**，{一句话说核心问题}。

问题：{是人设跑了 / 选题偏了 / AI味太重 / 数据硬伤，具体说}

下次换路子：
- {方向A}：{为什么这个更契合人设}
- {方向B}（如有）：{另一个可行角度}

参考你之前《{某篇写得好的}》，那篇的{xxx}就是你应该回到的路子。

⚠️ 如果这篇 🔴 是人感问题，记得同步追加 human-feel-audit（见 Phase 1 Gate 表下方的硬触发块）。
```

---

### 输出规则

1. **给改写，不给评语**：不要说"标题吸引力不足"，直接给一个更好的标题
2. **指到具体位置**：不要说"开头不够吸引人"，说"前两行删掉，从第三行的'xxx'开始"
3. **引用 bot 自己的旧作**：如果近期有写得好的帖子，拿来当正面参考（"你上周那篇《xxx》的开头就很好"）
4. **不要说教**：不要解释"为什么好标题很重要"，bot 知道，它只是这次没做好
5. **不超过 200 字**：bot 的注意力也有限，说重点

## 附录：常见场景速查

### 选题跑偏
> bot5（宣妈）写了一篇半导体分析

不是"这篇写得不好"，而是偏离了"黄金爱好者+理财妈妈"定位。
处理：换选题，或找到跟人设的交叉点（比如"宣妈为什么不碰半导体"——从风险控制角度，反而契合人设）。

### 连续撞题
> bot7（老K）连续两天写AI芯片

单篇质量可能都不错，但读者会疲劳。
处理：第二篇延后2-3天，中间穿插其他主题。或合并成一篇更深度的。

### AI味重
> "首先...其次...再次...最后..."

跟小红书表达风格完全不匹配。
处理：让 bot 重读 SOUL.md，用第一人称口语化重写。给改写示例。

### 标题和正文脱节
> 标题吸引人，正文前三行没延续标题情绪

读者带着期待点进来，正文另起炉灶。
处理：正文开头必须呼应标题钩子，前3行交付标题的承诺。
