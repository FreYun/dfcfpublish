## 一、主流记忆框架对比

### 1.1 框架概览

| 框架 | 开发者 | 核心理念 | 部署模式 | 开源/商业 |
|------|--------|---------|----------|-----------|
| **Letta** (原 MemGPT) | Letta Inc | 有状态 Agent，记忆分层（Core/Archival/Recall） | 自托管 Docker Server + SDK | 开源 + 商业云 |
| **Mem0** | Mem0 AI | 托管记忆层，User/Session/Org 三级记忆 | SaaS API / 开源自托管 | 开源 + 商业平台 |
| **Zep** | Zep Inc | Context Engineering 平台，时序知识图谱 | SaaS API / 开源 Graphiti | 开源 Graphiti + 商业 |
| **OpenClaw 原生** | OpenClaw | 文件系统记忆 (MEMORY.md + memory/ 日记) | 本地文件，零依赖 | 开源 |
| **skill_bag 记忆技能** | 社区 | Markdown 文件驱动的记忆系统 | 本地文件 + 可选向量搜索 | 开源 |

### 1.2 架构详解

#### Letta (原 MemGPT)

**核心机制**：
- **Core Memory**: 始终在 context window 中的关键信息（人物设定、用户偏好）
- **Archival Memory**: 向量数据库存储的长期记忆，通过 tool call 检索
- **Recall Memory**: 对话历史，支持搜索
- **Compaction**: 上下文窗口满时自动压缩，将重要信息提升到 Core/Archival

**技术栈**: Python Server + PostgreSQL + pgvector + Docker  
**接入方式**: REST API / Python SDK / MCP Server  

**优势**: Agent 自主管理记忆（读写 Core Memory 是 tool call），最接近"有记忆的 Agent"  
**劣势**: 需要独立部署服务器，架构复杂，与现有系统耦合度高

#### Mem0

**核心机制**：
- **Conversation Memory**: 当前轮次的短期上下文
- **Session Memory**: 单次任务/会话级别，任务结束后清理
- **User Memory**: 长期用户偏好和事实，跨会话持久
- **Org Memory**: 组织级共享知识

**技术栈**: Python SDK + 后端向量存储 + Graph Memory  
**接入方式**: `memory.add()` / `memory.search()` API 调用  

**优势**: API 简洁，分层清晰，支持 Graph Memory  
**劣势**: SaaS 依赖外部服务（数据出境风险），自托管需要向量数据库

#### Zep

**核心机制**：
- **Temporal Knowledge Graph**: 时序知识图谱 (Graphiti)，节点有时间戳
- **Facts**: 精确的关系事实，带时间戳
- **Summaries**: 实体历史摘要，随新信息更新
- **Context Assembly**: 查询时自动组装个性化上下文

**技术栈**: Graphiti (开源知识图谱) + Neo4j + Python  
**接入方式**: Python/TypeScript/Go SDK  

**优势**: 知识图谱适合复杂关系推理，时序特性支持"记忆老化"  
**劣势**: Neo4j 依赖重，运维复杂，对简单记忆场景过于 overkill

## 二、当前 OpenClaw 记忆现状

### 2.1 现有记忆机制

```
workspace-{agent}/
├── MEMORY.md           # 长期记忆（主会话注入 context）
├── SOUL.md             # 人格/行为准则（不变或缓慢变化）
├── IDENTITY.md         # 身份信息
├── memory/             # 日记文件
│   ├── YYYY-MM-DD.md   # 每日记录
│   └── topic-xxx.md    # 专题笔记
└── HEARTBEAT.md        # 心跳巡检清单
```

**特点**:
- ✅ 零依赖，纯文件系统
- ✅ 人类可读可编辑
- ✅ 每个 Agent 独立 workspace
- ❌ 无跨 Agent 记忆共享机制
- ❌ 无自动记忆提取/衰减
- ❌ 无语义搜索（只能全文匹配）
- ❌ 记忆容量受 context window 限制（MEMORY.md 写多了会被截断）

### 2.2 各 Agent 记忆使用情况

| Agent | MEMORY.md | memory/ 日记 | 评价 |
|-------|-----------|-------------|------|
| bot_main (魏忠贤) | ✅ 有 | ✅ 活跃（每日更新） | 使用最充分 |
| bot1 | ✅ 有 | ✅ 有子目录 (diary/xiaohongshu) | 结构化较好 |
| bot3 | ✅ 有 | ✅ 有日记 | 基本使用 |
| coder (工部) | ❌ 空 | ❌ 空 | 几乎未使用 |

## 三、skill_bag 中已有记忆技能评估

### 3.1 技能对比矩阵

| 技能 | 版本 | 记忆类型 | 存储方式 | 衰减机制 | 多Agent支持 | 复用可行性 |
|------|------|---------|---------|---------|------------|-----------|
| **memory-self-improving** | 1.0.11 | 学习日志 (errors/learnings/features) | Markdown 文件 | 无 | 有 (提升到 AGENTS.md) | ⭐⭐⭐⭐⭐ 直接可用 |
| **memory-认知记忆** | 1.0.8 | 四存储架构 (episodic/semantic/procedural/vault) | Markdown + JSON + Git | 指数衰减模型 | 共享读/受控写 | ⭐⭐⭐⭐ 需适配 |
| **memory-Agent 进化论** | 1.14.0 | 进化日志 (GEP 基因协议) | JSONL + Markdown | 无 | 无 | ⭐⭐ 概念好但风险高 |
| **memory-ontology** | 0.1.2 | 知识图谱 (实体+关系) | JSONL 图谱文件 | 无 | 可共享 | ⭐⭐⭐ 适合结构化知识 |
| **memory-thinking-model** | - | 思维模型框架 | 无持久存储 | 无 | 无 | ⭐⭐ 概念性质，非存储型 |
| **memory-Idea 教练** | - | 创意记录 | Markdown | 无 | 无 | ⭐ 场景单一 |

### 3.2 重点技能分析

#### memory-self-improving-1.0.11 — 推荐直接采用 ✅

**优势**:
- 最成熟的技能（1.0.11 版本，持续迭代）
- 与 OpenClaw workspace 完美兼容（直接写入 .learnings/）
- 有 Hook 集成（UserPromptSubmit / PostToolUse）
- 有提升机制（learnings → CLAUDE.md / AGENTS.md / SOUL.md）
- 有技能提取机制（重复 learning → 新 skill）

**接入成本**: 极低
```bash
# 每个 bot workspace 创建目录
mkdir -p ~/.openclaw/workspace-botN/.learnings
# 在 AGENTS.md 中加入引用
# 可选：配置 Hook
```

**预期收益**: 
- 各 bot 自动积累运营经验
- 错误模式识别和预防
- 逐步形成 bot 特定的最佳实践

#### memory-认知记忆 — 推荐改造后采用 ✅

**优势**:
- 最完整的记忆架构（四存储 + 衰减 + 反思）
- 支持多 Agent 共享读取
- 有初始化脚本和升级脚本
- 有审计追踪（Git + audit.log）

**需要改造的部分**:
1. **反思流程过重** — "请求 Token → 审批 → 内部独白"对 bot 场景不适用，需简化
2. **哲学元素** — "存在主义问题""暗色幽默"等对工具 bot 无意义，需移除
3. **衰减参数** — λ=0.03 (23 天半衰期) 需要根据我们的场景调整
4. **多 Agent 写入** — 当前是"主 Agent 写，子 Agent 提议"，需要改成各 bot 独立写

**接入成本**: 中等（需要 1-2 天改造）

#### memory-ontology-0.1.2 — 可选采用

**适用场景**: 需要跨 bot 共享结构化知识时（如用户画像、项目依赖关系）  
**接入成本**: 低  
**风险**: JSONL 图谱在数据量大时查询效率低

#### memory-Agent 进化论 — 不推荐 ❌

**原因**: 
- "自动修改自身代码"风险过高
- GEP 基因协议过于复杂
- 缺少安全防护机制
- 与我们"皇上说了算"的管理风格冲突

## 五、落地建议与优先级

### 5.1 推荐方案: 渐进式三阶段

#### 🟢 Phase 1: 立即可做（1-2 天）— self-improving 集成

**做什么**:
1. 为 bot1/3/4/5/6/7 的 workspace 创建 `.learnings/` 目录
2. 在各 bot 的 AGENTS.md 中追加 self-improving 引用块
3. 可选：配置 Hook 自动触发

**预期效果**:
- 各 bot 开始积累运营经验和错误模式
- 重复问题自动识别并提升为 AGENTS.md 规则
- 形成 bot 特定的最佳实践库

**风险**: 极低

#### 🟡 Phase 2: 短期改造（1 周）— 认知记忆精简版

**做什么**:
1. 基于 memory-认知记忆 改造精简版：
   - 保留：四存储架构、衰减模型、审计追踪
   - 移除：反思独白、Token 请求流程、哲学元素
   - 改造：多 bot 独立写入、衰减参数调优
2. 为 bot_main (魏忠贤) 先行试点
3. 验证效果后推广到其他 bot

**预期效果**:
- 重要记忆自动持久化，不重要的自然衰减
- 情景记忆 (episodic) 替代当前的简单日记
- 语义记忆 (semantic) 存储用户画像和项目知识
- 程序记忆 (procedural) 存储已验证的工作流

**风险**: 低

#### 🔴 Phase 3: 中期评估（1-2 月后）— 是否需要向量搜索

**评估条件**:
- 如果 Phase 1+2 后，各 bot 的记忆文件累计超过 50KB → 考虑引入向量搜索
- 如果需要跨 bot 记忆共享成为刚需 → 考虑 Mem0 自托管
- 如果不需要 → 继续用文件系统，省钱省心

**决策点**: Phase 2 上线 1 个月后复盘

### 5.2 不推荐的方案

| 方案 | 不推荐原因 |
|------|-----------|
| Letta 全量集成 | 架构改动过大，与 OpenClaw 耦合深，运维成本高 |
| Zep/Graphiti | Neo4j 太重，我们的场景不需要复杂的知识图谱 |
| Mem0 SaaS | 数据出境风险，依赖外部服务 |
| memory-Agent 进化论 | 自动修改代码风险过高，缺少安全防护 |

### 5.3 优先级总结

```
P0 (本周)  — self-improving 集成到所有活跃 bot
P1 (下周)  — 认知记忆精简版在 bot_main 试点
P2 (本月)  — 认知记忆推广到全部 bot
P3 (下月)  — 评估是否需要向量搜索/Mem0
```

## 六、总结

**核心结论**: 对于 OpenClaw 多 bot 体系，**不需要引入重型外部框架**。skill_bag 中已有的 `memory-self-improving` 和 `memory-认知记忆` 经过精简改造后，能满足 90% 的记忆需求，且保持了零依赖、人类可读、与 OpenClaw 原生兼容的优势。

**ROI 最高的动作**: 用 1-2 天时间集成 self-improving，让各 bot 立即开始积累经验。这一步的投入产出比远高于任何外部框架的接入。

**关键原则**: 记忆系统应该是**渐进式**的——先用简单方案解决 80% 的问题，等数据量和需求增长后再评估是否需要升级。不要过早引入复杂基础设施。
