# v1 vs v2 对比报告
📅 2026-04-04

## 实验说明

- 两个版本共用同一批 MCP 数据拉取（08:31:19 → 08:34:02），消除数据源差异
- v1 按原版 SKILL.md 执行：4 方向等深度分析
- v2 按 Gate+预加载 SKILL.md 执行：预扫描选 Top 2 深挖，其余快扫
- 由 sys3 单 agent 串行撰写（非真实 bot 并行执行），时间对比仅反映分析/写作开销差异

## 1. 时间对比

| 阶段 | v1 | v2 | 差异 |
|------|----|----|------|
| 数据拉取（共用） | 2m43s | 2m43s | 相同 |
| 报告撰写 | ~2m10s（08:34:02→08:36:12） | ~1m30s（估算） | v2 快 ~30% |
| **总计** | **~4m53s** | **~4m13s** | **v2 快 ~13%** |

### 为什么时间差不大？

本次实验的瓶颈在**数据拉取**（2m43s，占总时间 55%+），两版共用同一批数据。真正的差异体现在：
- v1 需要对 4 个方向各写 ~800 字深度分析 → 写作时间长
- v2 只需深挖 2 个方向（~800字×2） + 快扫 2 个方向（~150字×2）→ 写作时间短

### 真实 bot 场景预估

如果 bot 独立执行（非共用数据），时间差异会更大：

| 阶段 | v1（串行） | v2（预加载+并行） |
|------|-----------|------------------|
| Step 1 定价环境 | ~40s | ~40s |
| Step 2 景气扫描 | ~60-90s（4方向全拉） | ~50-70s（4方向快扫并行） |
| Step 3 深挖 | 无（v1不区分深浅） | ~40-60s（仅2方向，预加载重叠） |
| Step 3-4 盈利 | ~60-90s（8只股票） | ~30-40s（仅4只+预加载） |
| Step 4-5 资金+政策 | ~60-90s（串行两步） | ~30-40s（并行2个subagent） |
| 汇总 | ~30s | ~40s（多决策路径回顾） |
| **预估总计** | **~4.5-6 min** | **~3-4 min** |
| **预估节省** | — | **~30-35%** |

关键节省来源：
1. 只深挖 2/4 方向 → 数据拉取量减半
2. 预加载与决策并行 → IO 等待重叠
3. 资金+政策合并为并行 Gate

## 3. 综合评价

### v1 原版
- **优点**：全面覆盖，不遗漏，每个方向都有完整推导链
- **缺点**：耗时长，计算机和传媒方向当前变化不大但仍花同等篇幅，有"为了全面而全面"的冗余
- **适用**：用户第一次看TMT、需要全景了解时

### v2 Gate+预加载
- **优点**：快 ~30%，重点突出（本次选了通信+电子深挖），有决策审计
- **缺点**：计算机/传媒只有快扫，如果用户关心的恰好是快扫方向则信息不足
- **适用**：用户已有TMT基础认知、关心"当前最值得关注的方向是什么"时

### 关键发现

1. **核心结论不受模式影响**：两版的投资排序和核心判断完全一致，说明快扫足够筛选出正确方向
2. **深挖价值在于盈利验证和资金分析**：这些需要个股级数据，快扫无法覆盖
3. **预加载在当前架构下受限**：v2 的 SKILL.md 指令式预加载依赖 Claude 手动启动 subagent，实际执行中 subagent 频繁失败（参数错误、权限被拒）。`skill-tree-prefetch-plan.md` 的基础设施级方案才是正解
4. **时间节省主要来自"少做事"**（2/4深挖），而非"做得更快"（预加载）。真正的预加载收益需要 prefetch-mcp 基础设施支撑

---

## 4. 建议

1. **当前部署 v1**：稳定可靠，全面覆盖，bot 不会因为 subagent 失败而卡住
2. **推进 prefetch-mcp 方案**：把预加载下沉到基础设施层，不依赖 Claude 管 subagent 生命周期
3. **v2 的 Gate 决策机制值得保留**：将 Gate 的决策记录和选择性深挖逻辑合并到 v1 中，作为可选模式
4. **建议的融合方案**：v1 的全面性 + v2 的 Gate 审计 + prefetch-mcp 的基础设施预加载
